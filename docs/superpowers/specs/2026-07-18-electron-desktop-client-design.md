# Electron 跨平台客户端设计

## 背景

项目当前的迁移能力由 Python 实现，Flask 提供 Web 页面和接口，macOS 客户端使用 PyWebView 将 Flask 页面装进桌面窗口。本次改造采用 Electron，并将渲染层重写为 React + Vite + TypeScript，同时继续复用 Flask/Python 后端和现有迁移业务逻辑。

目标平台为 macOS 和 Windows。应用面向不一定预装 Python 的桌面用户，启动、选择文件、配置 Vault、执行迁移和查看结果都应在一个应用窗口内完成。

## 目标与非目标

### 目标

- 使用 Electron 提供 macOS 和 Windows 的统一桌面窗口、单实例和生命周期管理。
- 使用 React + Vite + TypeScript 重写前端交互，不再把 Jinja/Bootstrap 页面作为桌面客户端的主界面。
- 复用现有 Flask/Python 的 Evernote 导出、ENEX 处理、Markdown 转换、Obsidian 写入和迁移进度能力。
- 通过打包后的 Python 子进程提供本机服务，用户无需单独安装 Python 或手动启动 Flask。
- 采用单窗口向导，覆盖数据源选择、目标配置、预检、迁移和结果查看。
- 支持 macOS 和 Windows 的构建、安装和运行时冒烟验证。
- 对后端启动失败、迁移部分失败、窗口关闭和重复启动提供可理解的反馈。

### 非目标

- 不在本次工作中重写迁移算法或替换现有 Python 业务模块。
- 不实现云端迁移服务、远程任务队列或账号托管。
- 不把系统能力直接暴露给 React 页面；文件和窗口能力必须经由受限 preload API。
- 不在本次工作中承诺 Apple notarization、Windows 代码签名或自动更新；这些属于发布配置。
- 不保留 Jinja/Bootstrap 作为 Electron 客户端的长期渲染方案。旧 Web 页面可在迁移期间保留作为兼容入口。

## 运行前提与操作指南

开发或发布此客户端时，在执行任何 `npm` 安装、测试、构建或打包命令前必须使用 **Node.js >=22.12.0**。npm 的 engine 警告不足以阻止不受支持的 Node 版本继续安装或构建，因此不能把警告当作兼容性检查；发布打包必须使用 Node 22.12+。已安装的桌面应用不要求终端用户预装 Node.js。当前可执行的开发、构建、打包和故障排除命令以 [ELECTRON_CLIENT.md](../../../ELECTRON_CLIENT.md) 为准；本设计保留架构决策与验收范围。

## 方案选择

### 方案 A：Electron 仅替换 PyWebView

Electron 只负责启动 Python 服务并加载现有 Flask 页面。优点是初始改造最快，缺点是无法充分改善单窗口交互，桌面文件能力和错误状态仍受原有页面结构限制。

### 方案 B：Electron 外壳加受控复用

保留现有页面，只增加 Electron 的进程管理、健康检查和少量 preload 能力。风险低，但前端技术债和多页面流程仍然存在。

### 方案 C：Electron + React 前端重写（采用）

Electron 主进程管理桌面生命周期，React 渲染层负责完整的单窗口向导，Flask/Python 只提供迁移服务和进度事件。该方案能复用高价值的 Python 业务逻辑，同时建立清晰的桌面前端边界，适合后续增加预检、恢复和结果分析能力。

## 交互设计

### 主流程

主窗口使用顶部步骤条，流程固定为：

1. **选择数据源**：选择 Evernote 数据文件或目录，校验路径并显示可识别的数据量。
2. **配置目标**：选择 Obsidian Vault，设置附件处理、文件命名和重复文件策略。
3. **迁移预检**：展示源数据、目标位置、预计数量和风险提示，用户确认后开始迁移。
4. **执行迁移**：展示总体进度、当前处理项、成功/跳过/失败数量和可展开实时日志。
5. **迁移结果**：展示结果摘要、失败项和警告，提供打开 Vault、查看日志和重新开始等操作。

页面之间前进和后退时保留表单状态。迁移开始后锁定配置步骤，避免用户在任务运行期间修改目标。

### 窗口布局

- 顶部放置应用名称、当前步骤条和帮助/设置入口。
- 中部放置当前步骤的主要内容，桌面宽度下最大约 960–1100px。
- 底部使用固定操作栏，统一放置上一步、下一步和开始迁移等操作。
- 迁移中采用进度与统计区域加可折叠日志区域的组合布局。
- 结果页先展示成功、警告、失败数量，再提供明细和后续操作。
- 建议最小窗口尺寸为 960×640，支持 macOS 和 Windows 常见桌面分辨率。
- 颜色只表达状态语义：蓝色表示进行中，绿色表示成功，黄色表示警告，红色表示失败；所有关键状态同时使用文字或图标表达。
- 所有控件支持键盘焦点和基本键盘操作，避免只依赖颜色或鼠标。

### 状态与异常

前端状态流转为：

`启动中 → 可配置 → 预检中 → 可迁移 → 迁移中 → 成功 / 部分成功 / 失败`

关键行为如下：

- Python 后端启动失败时显示诊断信息和重新连接操作。
- 数据源无效、为空或格式不支持时在选择页直接提示原因。
- Vault 不可写时阻止进入迁移，并提示检查权限或路径。
- 预检发现风险时允许返回修改，也允许用户确认后继续。
- 迁移部分失败时分别展示成功、跳过和失败数量，并保留失败日志。
- 迁移过程中关闭窗口时提示：已写入内容会保留，当前任务将停止；确认后执行安全停止。
- 重复启动应用时不创建第二个窗口，而是聚焦已有窗口。
- 错误信息必须包含可执行的下一步，不只显示异常文本。

日志分为页面内实时日志和本地完整日志文件。实时日志用于判断当前进度，完整日志用于排查和反馈问题。

## 总体架构

```text
┌─────────────────────────────────────────────┐
│ Electron 主进程                              │
│  单实例、窗口、Python 子进程、端口、退出清理  │
└───────────────┬─────────────────────────────┘
                │ 受限 preload API
┌───────────────▼─────────────────────────────┐
│ React Renderer                               │
│  向导状态、表单、进度、日志、结果              │
└───────────────┬─────────────────────────────┘
                │ HTTP + Socket.IO
┌───────────────▼─────────────────────────────┐
│ Python Flask / Flask-SocketIO 子进程          │
│  迁移 API、任务、导出、转换、Vault 写入         │
└─────────────────────────────────────────────┘
```

### Electron 主进程

主进程负责确定打包前后的资源路径，启动 Python 后端，分配 `127.0.0.1` 动态端口，轮询健康检查，并在服务就绪后创建唯一主窗口。窗口加载本地 React 构建产物，不能导航到未授权的外部地址。

关闭窗口时，主进程先请求后端安全停止当前任务并等待短暂的清理窗口；超时后终止子进程，确保不遗留独立的 Flask 服务。应用启动参数、后端标准输出和退出码应保留在诊断日志中。

### React 渲染层

React 使用单一向导状态模型，而不是通过多个页面 URL 表达流程。建议按职责拆分为：

- `wizard`：步骤状态、前进/后退和表单数据。
- `source`：数据源选择、路径校验和数据摘要。
- `target`：Vault 和迁移策略配置。
- `preflight`：预检结果、风险和确认。
- `migration`：任务启动、Socket.IO 进度和实时日志。
- `result`：结果统计、失败明细和打开路径操作。
- `shared`：按钮、表单、状态提示、错误面板和设计变量。

渲染层只通过 HTTP 客户端和 `socket.io-client` 与后端通信。需要访问文件选择器、打开路径、获取应用信息或请求退出时，使用 preload 暴露的白名单方法，不直接启用 Node.js。

### Python 后端

现有迁移器、导出器、Markdown 转换器和 Vault 写入逻辑保持为后端职责。现有接口可以作为兼容基础，包括迁移启动、ENEX 导出、ENEX 上传、迁移状态查询和 Socket.IO 任务房间；桌面前端接入时应统一返回机器可读的错误码、用户可读的消息和可选诊断信息。

为 Electron 客户端补充或规范以下能力：

- `/api/healthz`：报告进程已启动且 API 可用。
- 预检接口：只读检查输入、目标路径、权限、重复策略和可迁移数量。
- 迁移启动和状态接口：返回任务 ID、阶段、统计和最终结果。
- 安全停止接口：在任务边界停止当前任务，并明确已写入文件不会回滚。
- Socket.IO 进度事件：至少包含任务 ID、阶段、进度、当前项、统计、日志和错误信息。

现有 Jinja 页面和路由在前端迁移期间可以保留，用于回归和兼容；Electron 客户端不依赖这些模板作为主界面。

## 启动、端口与安全

1. Electron 主进程解析开发环境和打包环境的后端资源路径。
2. 绑定 loopback 临时端口，避免依赖固定的 5000 端口。
3. 以参数或环境变量启动 Python 后端，并明确传入 `127.0.0.1` 和动态端口。
4. 等待 `/api/healthz` 返回成功，超时则展示后端启动错误页。
5. 将后端地址通过 preload 的最小 API 提供给 React。
6. React 使用该地址建立 HTTP 和 Socket.IO 连接。
7. 应用退出时停止任务、关闭窗口并清理 Python 子进程。

后端只监听 `127.0.0.1`，Socket.IO 的跨域配置应收敛到 Electron 本地渲染来源和开发来源，而不是长期保留任意来源。Electron 开启 context isolation，页面不启用 Node integration，并通过 CSP、导航拦截和外部链接策略限制内容边界。

## 打包与平台支持

React 使用 Vite 生成 renderer 资源，Electron 使用 `electron-builder` 生成安装产物。Python 后端在目标平台分别使用 PyInstaller 构建，作为 Electron 的 `extraResources` 放入安装包：

- macOS：按现有架构生成 `.app`，再生成 `.dmg`；发布签名和 notarization 作为后续发布配置。
- Windows：生成 x64 后端可执行文件和 Electron 安装包；处理 Windows 路径、权限和进程终止语义。
- 开发环境：允许 Electron 直接启动仓库中的 Python 模块，避免每次改动都重新打包后端。
- 生产环境：只启动安装包内与当前平台匹配的后端可执行文件。

构建流水线必须在 macOS 和 Windows 上分别执行，不能假设一个平台的 PyInstaller 产物可跨平台运行。构建产物应包含前端 bundle、Electron 主进程、preload、Python 后端及其模板/静态资源依赖。

## 测试与验收

### React

- 向导状态流转、前进/后退和表单保留。
- 数据源、Vault、策略和预检表单的校验。
- 启动中、迁移中、成功、部分成功和失败状态渲染。
- Socket.IO 进度事件到统计和日志的映射。
- 错误面板和关闭确认行为。

### Electron

- 动态端口分配和健康检查重试。
- 开发/打包资源路径解析。
- Python 子进程启动、异常退出和安全清理。
- 单实例锁和重复启动时聚焦已有窗口。
- preload 只暴露白名单能力，页面不能直接访问 Node.js。

### Python/API

- 健康检查、预检、迁移启动、状态查询和停止接口。
- 错误码、任务 ID、结果统计和进度事件结构稳定。
- 既有迁移业务测试继续运行；网络账号和真实 Vault 测试按仓库当前约束单独执行。

### 打包验收

- macOS 产物可启动，能连接本机后端并打开首页流程。
- Windows 安装包可安装、启动、选择路径并完成最小迁移冒烟测试。
- 应用退出后没有遗留 Python 后端进程。
- macOS 和 Windows 的路径、临时目录、日志目录和打开 Vault 操作均有效。

## 风险与应对

- **动态端口与本地来源通信**：统一由主进程提供后端地址，并为开发和生产来源配置明确的 CORS 与 Socket.IO 规则。
- **文件路径差异**：路径选择和打开操作经 preload 处理，后端统一使用 `pathlib`，测试覆盖 Windows 分隔符和盘符。
- **打包资源缺失**：将前端、preload、Python 可执行文件、模板和静态资源分别列入构建检查，并执行安装后冒烟测试。
- **迁移不可回滚**：结果页和关闭确认明确说明已写入内容；停止接口只保证安全停止，不承诺撤销已有文件。
- **旧页面行为与新前端不一致**：先以现有后端 API 和真实迁移结果为验收依据，再逐步下线 Electron 不再使用的 Jinja 路由。
